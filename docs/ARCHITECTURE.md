# Архитектура проекта

Этот документ — **источник правды по архитектуре и ключевым решениям** конкретного проекта: модули, границы ответственности, потоки данных, интеграции, соглашения по коду и развёртыванию. По мере развития проекта сюда добавляются новые разделы и уточнения.

## Область действия этого документа

Этот файл и каталог `docs/` относятся **только к тому корню папки в Cursor**, в котором они находятся — в терминах Cursor это **один из корней workspace** (при multi-root — отдельный «проект» в боковой панели), а не весь состав окна целиком.

При работе с несколькими открытыми корнями изменения в архитектуре и документации вносятся **только здесь**, без смешивания с файлами соседних корней.

*Примечание: чаще всего этот корень совпадает с корнем одного Git-репозитория; если структура другая — областью действия остаётся именно эта папка.*

Далее зафиксированы **условия локальной работы и синхронизации с продакшеном** на машине разработчика. Это не описание доменной архитектуры приложения, а общая рамка, в которой ведётся разработка **данного корня проекта**.

## Локальное окружение и рабочие копии

### Каталоги

#### Разработка (локальная рабочая копия)

- **Базовый путь:** `C:\cursors\dev`
- **Проект:** `C:\cursors\dev\<ИМЯ_ПАПКИ_ПРОЕКТА>`

Здесь ведётся основная работа в редакторе, коммиты и ветки Git (если репозиторий настроен).

#### Продакшен / серверное зеркало

- **Базовый путь:** `P:\`
- **Проект:** `P:\<ИМЯ_ПАПКИ_ПРОЕКТА>`

Каталог на диске `P:\` является зеркалом, синхронизируемым с сервером.  
Изменения в папке `P:\` отражаются в серверной копии проекта.

### Синхронизация `P:\` с сервером

- Тип синхронизации: **папка Google Drive**
- `P:\` используется как синхронизируемое зеркало серверных файлов

### Схема базы данных (выгрузка на зеркале)

На зеркале продакшена по пути **`P:\<ИМЯ_ПАПКИ_ПРОЕКТА>\sql\db_schema.sql`** хранится **актуальная на момент генерации схема** таблиц проекта (DDL / структура).

- Файл **обновляется автоматически раз в сутки**.
- Указанная актуальность относится к **состоянию на предыдущий календарный день** (снимок не «прямо сейчас», а по расписанию суточного цикла).

Для разработки опираться на этот файл как на **справочник структуры**; при расхождениях с кодом или миграциями приоритет имеют принятый в проекте источник правды (миграции, прод после деплоя и т.п.) — уточните при необходимости отдельно.

### Политика редактирования

- **Напрямую редактировать файлы на `P:\` запрещено**
- Все изменения вносятся в локальной dev-копии: `C:\cursors\dev\<ИМЯ_ПАПКИ_ПРОЕКТА>`

### Перенос изменений между `C:\cursors\dev` и `P:\`

Между локальной dev-копией и каталогом на `P:\` используются два подхода:

1. **Git (основной способ, если настроен в проекте)**  
   Изменения переносятся через стандартный Git-поток (commit/push/pull/merge по принятому процессу).

2. **Ручное копирование**  
   Если Git не настроен для проекта — допускается ручное копирование файлов между `C:\cursors\dev\<ИМЯ>` и `P:\<ИМЯ>`.

### Git-конвенция по умолчанию

- Во всех проектах, где настроен Git, по умолчанию используется ветка **`main`**
- Исключение: если для конкретного проекта/задачи явно задана другая ветка

### Краткие правила

| Где | Назначение |
|-----|------------|
| `C:\cursors\dev\...` | Основная локальная разработка |
| `P:\...` | Зеркало серверной копии через Google Drive |
| Редактирование на `P:\` | Запрещено |
| Между `C:\cursors\dev` и `P:\` | Git конкретного проекта или ручное копирование |
| Git-ветка по умолчанию | `main` |

## Документация по скриптам и реализации

Общие решения, границы модулей и потоки данных фиксируются в этом файле (**ARCHITECTURE.md**).

Подробности реализации отдельных скриптов — классы, функции, форматы входов/выходов, краевые случаи, примеры запуска — выносим в отдельные файлы в каталоге **`docs/`**:

- **Имя файла:** `docs/<имя_скрипта>.md`, где `<имя_скрипта>` совпадает с именем скрипта **без расширения** (например, для `deploy.ps1` → `docs/deploy.md`, для `sync_data.py` → `docs/sync_data.md`).
- **Содержание:** всё, что нужно для понимания и сопровождения именно этого скрипта; дублировать общую архитектуру приложения здесь не обязательно — достаточно ссылки на соответствующий раздел **ARCHITECTURE.md**, если нужен контекст.

## LLM Connector (`llm_connector` pip package)

Отдельный репозиторий: `C:\cursors\dev\_all_llm-connector` → зеркало `P:\_all_llm-connector`.

### Назначение

- Маршрутизация LLM-вызовов по таблицам `llm_*` в MySQL
- Логирование в `llm_request_logs`
- Multi-stage fallback (same-provider model → cross-provider)
- Recovery частичных ответов в `runtime/llm_recovery/` (потребитель)
- MVP: режим **internal / normal** — один DSN primary, без outbox

### Таблицы

| Таблица | Роль |
|---------|------|
| `llm_projects` | Код проекта-потребителя (`ailenta_parser`) |
| `llm_providers` | OpenRouter, VseGPT, … + `shared_api_key_env` |
| `llm_routes` | `caller_script` + `function_key` + `model_slot` |
| `llm_request_logs` | Аналитика вызовов |

**Физически:** отдельная БД **`_llm_connector`** на том же MySQL-сервере (не в `ailenta_parser`).

Миграции: `docs/migrations/000_create_database.sql` → `001` → `002`.

### Фаза 2 (заглушки)

- `llm_connector.storage.resilient.ResilientStorage` — replica + outbox
- `scripts/sync_outbox_to_primary.py` — replay на primary
- Env: `LLM_DEPLOYMENT_CODE=external`, `LLM_DB_MODE=auto`

### Интеграция в ailenta_parser

```bash
pip install -r requirements_llm_connector.txt
```

Мост: `pys/llm_connector_bridge.py` (только project_code + recovery path).
DSN и подключение: `llm_connector.db_config` / `llm_connector.db_connection` + env `LLM_DB_*`.
Пилот: `prompts_pre_list_headers_vote.py`, `prompt_meta_extract.py`.
